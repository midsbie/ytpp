* ytpp

Post-process YouTube downloads for TV compatibility.

Older smart TVs (e.g. LG C8) can't decode AV1 video or Opus audio, the codecs that YouTube increasingly defaults to.  =ytpp= re-encodes these streams to H.265/AC3 while copying anything already compatible, so videos play without hassle.

** Usage

#+begin_example
ytpp [options] <file>
#+end_example

| Option               | Description                                                        |
|----------------------+--------------------------------------------------------------------|
| =-q, --quality <n>=  | Quality level: QP for NVENC/VAAPI, CRF for software (default: 17)  |
| =-f, --force=        | Force transcoding even if codecs are already compatible             |
| =--sw=               | Force software encoding (libx265)                                  |
| =--no-delete=        | Keep the original file after conversion                            |
| =-h, --help=         | Show help                                                          |

** How it works

1. Probe the input for codec and dimension info.
2. Re-encode heuristics:
   - If AV1: re-encode video to H.265.
   - If Opus: re-encode audio to AC3 (Dolby Digital).
   - Otherwise: streams are considered compatible and copied as-is.
3. Video encoder is selected automatically: NVENC > VAAPI > software (via libx265).  Use =--sw= to force software encoding.

*** NVENC

When an NVIDIA GPU is available, =ytpp= uses =hevc_nvenc= with preset =p7= (highest quality), =hq= tune, B-frame references, spatial AQ, and 10-bit output.

*** VAAPI (Mesa radeonsi caveats)

VAAPI is used as a fallback when no NVIDIA GPU is available.  The Mesa radeonsi driver appears to have issues with HEVC encoding:

1. *Green bar artifact:* AMD VCN seems to require surface heights aligned to 32 (or 64×16 depending on generation).  Non-aligned heights (e.g. 1080) may be padded to the next aligned value (1088), filling extra rows with uninitialized data that shows up as a green or noise bar.  =ytpp= works around this by scaling *up* to the aligned height in software, uploading to VAAPI, then scaling back *down* with =scale_vaapi=.  Already-aligned heights (720p, 2160p) skip the extra scaling.

   This appears to involve Mesa, libva, and FFmpeg, as radeonsi may have historically reported incorrect block sizes, and FFmpeg defaulted to 16×16 alignment.  An upstream FFmpeg [[https://www.mail-archive.com/ffmpeg-devel@ffmpeg.org/msg161546.html][patch]] (Mar 2024) adds =VASurfaceAttribAlignmentSize= support, requiring libva ≥ 1.21.0.

   - [[https://gitlab.freedesktop.org/mesa/mesa/-/issues/9196][Mesa #9196]] :: Incorrect HEVC block size reported with radeonsi
   - [[https://gitlab.freedesktop.org/mesa/mesa/-/issues/9185][Mesa #9185]] :: Encoded files include padding at bottom/right (HEVC, AV1)
   - [[https://gitlab.freedesktop.org/mesa/mesa/-/issues/4559][Mesa #4559]] :: hevc_vaapi: wrong image width (48 px black bar)
   - [[https://bugs-devel.debian.org/cgi-bin/bugreport.cgi?bug=1081512][Debian #1081512]] :: hevc_vaapi produces 1920×1088 with green bar on AMD (Sep 2024)
   - [[https://github.com/obsproject/obs-studio/pull/11998][OBS Studio #11998]] :: Fix incorrect HEVC vertical resolution on AMD VCN (Apr 2025)
   - [[https://github.com/GPUOpen-LibrariesAndSDKs/AMF/issues/423][AMF #423]] :: AMD acknowledges alignment as a VCN4 limitation (Oct 2023)
   - [[https://github.com/LizardByte/Sunshine/issues/4314][Sunshine #4314]] :: Green line / black bars with HEVC on AMD (Sep 2025)

2. *Intermittent frame corruption:* the encoder may produce malformed frames at unpredictable points.  On TVs this can manifest as audio stuttering and resync pauses (possibly the hardware decoder stalling on a bad frame); in VLC it may show as visual corruption.

   - [[https://gitlab.freedesktop.org/mesa/mesa/-/issues/9436][Mesa #9436]] :: VCN2: broken H.264 video, HEVC encoder crashes
   - [[https://gitlab.freedesktop.org/mesa/mesa/-/issues/11730][Mesa #11730]] :: Tearing in all videos encoded via VAAPI
   - [[https://bugs.freedesktop.org/show_bug.cgi?id=109538][fd.o #109538]] :: Unstable HEVC encoding, garbled output (Feb 2019)
   - [[https://bugs.freedesktop.org/show_bug.cgi?id=110673][fd.o #110673]] :: HEVC encoding: segfault and garbled contents (May 2019)

NVENC appeared to avoid both issues in limited testing.

*** Software (libx265)

The software path uses 10-bit encoding (=yuv420p10le=) with =aq-mode=3= to redistribute bits toward dark scenes and reduce banding.

** ytdl

A convenience wrapper around =yt-dlp= for downloading is also included:

#+begin_example
ytdl <yt-dlp args>
#+end_example

It passes =--cookies= automatically when =cookies.txt= exists in the working directory, and adds =--js-runtimes node= when Node.js is available.  The subsections below cover general yt-dlp setup that =ytdl= benefits from but that applies to any direct =yt-dlp= usage as well.

*** Cookies

YouTube's anti-bot measures often require browser cookies for yt-dlp to work reliably.  =ytcks= extracts cookies into a file that =ytdl= picks up automatically:

#+begin_example
ytcks [browser]
#+end_example

Without an argument, =ytcks= tries to detect the default browser via =xdg-settings=.  Firefox users with multiple profiles may need to specify the profile path explicitly:

#+begin_example
ytcks "firefox:~/.mozilla/firefox/xxxxxxxx.profile-name"
#+end_example

This creates =cookies.txt= (=chmod 600=) in the current directory.  Set the =COOKIES= environment variable to override the path:

#+begin_example
COOKIES=~/cookies.txt ytcks firefox
COOKIES=~/cookies.txt ytdl <yt-dlp args>
#+end_example

*** PO Token (POT) provider

YouTube's bot detection may block downloads even with valid cookies.  When this happens, yt-dlp likely needs a Proof of Origin (PO) token provider plugin.

1. Install the [[https://github.com/Brainicism/bgutil-ytdlp-pot-provider][bgutil-ytdlp-pot-provider]] plugin by extracting the latest release ZIP under =~/.config/yt-dlp/plugins/=:

   #+begin_example
   ~/.config/yt-dlp/plugins/
   └── bgutil-ytdlp-pot-provider/
       └── yt_dlp_plugins/
           └── extractor/
               ├── getpot_bgutil_http.py
               ├── getpot_bgutil.py
               └── getpot_bgutil_script.py
   #+end_example

2. Start the POT provider container:

   #+begin_example
   docker run --name bgutil-provider -d -p 4416:4416 --init brainicism/bgutil-ytdlp-pot-provider
   #+end_example

3. Add =--js-runtimes node= to your yt-dlp config (=~/.config/yt-dlp/config=) so the plugin's JS evaluation uses Node.js.

*** Recommended yt-dlp config

#+begin_example
-4
--mark-watched
#+end_example

=-4= avoids IPv6-related 403 errors on reconnect.  =--mark-watched= marks videos as watched on YouTube after downloading (requires cookies; no-ops without auth).

*Note:* do not set =--extractor-args=, =-f=, or =--js-runtimes= here — =ytdl= passes its own values and yt-dlp CLI args override the config file.

** Dependencies

- =ffmpeg= / =ffprobe=
- =yt-dlp= (for =ytdl= only)
- =node= (for the POT provider plugin's JS evaluation)
